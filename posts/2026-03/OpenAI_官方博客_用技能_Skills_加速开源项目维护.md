<div><section><p><span leaf="">我们用 Codex 改变了维护 OpenAI Agents SDK</span><sup><span leaf="">[1]</span></sup><span leaf=""> 仓库的方式。仓库本地的技能（skills）、</span><code><span leaf="">AGENTS.md</span></code><span 个（Python：182 → 226，TypeScript：134 → 231）。</span></p><p><span leaf="">简单介绍一下背景：该 SDK 提供 Python</span><sup><span leaf="">[2]</span></sup><span leaf=""> 和 TypeScript</span><sup><span 只在需要时才读取参考资料或运行脚本</span></section></li></ul><p><span leaf="">两个 SDK 仓库都把这些工作流放在代码旁边：</span></p><ul><li><section><span leaf="">• openai-agents-python 中的 .agents/skills</span><sup><span leaf="">——将上一个发布标签与当前发布候选版本进行对比，检查发布就绪状态。</span></section></li><li><section><span leaf="">• </span><code><span leaf="">implementation-strategy</span></code><span leaf="">——在修改运行时或 API leaf="">技能要真正发挥作用，得在对的时机被强制触发。这就是 </span><code><span leaf="">AGENTS.md</span></code><span leaf=""> 的用武之地。</span></p><p><span leaf="">AGENTS.md 指南</span><sup><span leaf="">[11]</span></sup><span 应用。这抓的是另一类问题：不是“示例在仓库里能不能跑”，而是“包在经过发布、安装和运行时集成之后，行为是否仍然正确”。</span></p><p><span leaf="">合在一起看，就能理解为什么技能、脚本和模型判断的组合管用。脚本让运行可重复、捕获证据、覆盖手动检查起来繁琐的安装路径。Codex 则利用这些证据，做出比简单的脚本通过/失败检查更细致的对比。</span></p><h2 data-heading="true"><span leaf="">增加发布检查</span></h2><p><span leaf="">发布准备是这个模式特别合适的另一个场景。</span></p><p><span leaf="">两个仓库中的发布审查工作流从查找上一个发布标签开始，将其与最新的 </span><code><span leaf="">main</span></code><span leaf=""> 分支进行 diff，然后要求 Codex 检查该差异中的：</span></p><ul><li><section><span leaf="">• 公共 API 和面向用户的 SDK 行为中的向后兼容性问题</span></section></li><li><section><span leaf="">• 回归问题，包括预期行为的细微变化</span></section></li><li><section><span leaf="">• 需要迁移说明或发布说明更新但尚未添加的变更</span></section></li></ul><p><span leaf="">根据这些发现，技能做出整体的发布就绪性判断。</span></p><p><span leaf="">一个具体的例子是 openai/openai-agents-python#2480</span><sup><span leaf="">[23]</span></sup><span leaf="">，在这次发布审查中，整体结论是绿灯放行，同时指出了 Python 3.9 支持的移除以及它所需的发布说明跟进：</span></p><pre><span><svg leaf="">技能还定义了关卡决策的方式。审查从“可以安全发布”出发，只有当差异中出现了真实问题的具体证据时才切换为阻止发布。每个阻止决定都必须附带具体的解除阻止清单。这让输出更易使用：绿灯意味着差异中未发现阻止发布的问题，红灯意味着存在真实问题且有明确的下一步。</span></p><p><span leaf="">这比泛泛的“请审查一下这次发布”要有用得多。它迫使模型基于具体的差异进行推理，并用可操作的术语解释结果。如果发布是安全的，就直接说安全。如果不安全，就指出确切的证据和确切的后续步骤。</span></p><h2 data-heading="true"><span leaf="">在 CI 中运行工作流</span></h2><p><span leaf="">当一个技能在本地验证有用之后，Codex GitHub Action</span><sup><span leaf="">[6]</span></sup><span leaf=""> 可以轻松地把同一个工作流自动化到 CI 中。最好等本地工作流稳定了再上 CI，因为手动跑才是调试指令、打磨脚本、发现边界情况的过程。</span></p><p><span leaf="">对于公开仓库，触发器的设计和技能本身同样重要。GitHub Action 安全清单</span><sup><span leaf="">[24]</span></sup><span leaf="">建议限制谁能启动工作流、优先使用可信事件或显式审批、对来自 PR、提交、issue 或评论的 prompt 输入进行清理、用 </span><code><span leaf="">drop-sudo</span></code><span leaf=""> 或非特权用户保护 </span><code><span leaf="">OPENAI_API_KEY</span></code><span leaf="">，以及把 Codex 作为任务中的最后一步运行。</span></p><p><span leaf="">如果一个工作流具有写权限且接受不可信的公开输入，风险通常出在技能周围的触发器设计、输入处理和运行时权限上。</span></p><h2 data-heading="true"><span leaf="">在 PR 审查中使用 Codex</span></h2><p><span leaf="">技能是这些仓库效率提升的一部分。Codex GitHub PR 自动审查</span><sup><span leaf="">[25]</span></sup><span leaf="">是另一部分。</span></p><p><span leaf="">自从 Codex GitHub PR 自动审查上线以来，Codex 在这些仓库的大多数代码变更审查中都有帮助。我们把它当常规审查环节，不是特殊工具。</span></p><p><span leaf="">对于直接的程序错误、回归和缺失的测试，把 Codex 作为必要的审查路径在实践中已经足够可靠。它能始终如一地反复检查同样的正确性模式，并消除了小修复和日常改进的一个主要瓶颈。</span></p><p><span leaf="">人工审查依然重要，但针对的是另一类变更。</span></p><p><span leaf="">当核心问题不是“这段代码正确吗”而是“在多个合理的方案中我们该选哪个，以及该如何发布”时，人工审查仍然不可或缺。这包括：</span></p><ul><li><section><span leaf="">• API 或架构变更，存在多个合理的设计方案，维护者需要做出明确选择</span></section></li><li><section><span leaf="">• 行为变更，影响产品预期、向后兼容性承诺或发布策略</span></section></li><li><section><span leaf="">• 命名、迁移和发布沟通决策，难点在于选择对用户和贡献者最清晰的方案</span></section></li><li><section><span leaf="">• 需要维护者或团队间对齐的变更，比如确定范围、排列优先级，或决定哪些内容现在发布、哪些以后再说</span></section></li></ul><p><span leaf="">Codex 在所有这些场景中仍然能提供有用的贡献，但它们仍然需要人类决策者和直接讨论。</span></p><p><code><span leaf="">AGENTS.md</span></code><span leaf=""> 也可以编码这种分工：仓库可以告诉 Codex 在正确性审查中什么是重要的，Codex 就能持续一致地应用这些指导。</span></p><p><span leaf="">这也是效率提升的重要贡献者。重复性的审查和验证工作不再需要为每个低风险变更等待稀缺的审查者时间，而维护者可以专注于需要他们判断力的高上下文审查。这种转变帮助我们更快地处理了积压的 bug 和小规模的功能改进。</span></p><h2 data-heading="true"><span leaf="">写在最后</span></h2><p><span leaf="">在 OpenAI Agents SDK 仓库中，技能在融入仓库的日常工作配置时效果最好。</span></p><p><code><span leaf="">AGENTS.md</span></code><span leaf=""> 告诉 Codex 哪些工作流是必需的。</span><code><span leaf="">description</span></code><span leaf=""> 告诉它何时路由到这些工作流。</span><code><span leaf="">scripts/</span></code><span leaf=""> 处理确定性的部分。模型处理需要上下文的部分。而当一个工作流在本地稳定之后，Codex GitHub Action</span><sup><span leaf="">[6]</span></sup><span leaf=""> 可以把同样的流程带入 CI。</span></p><p><span leaf="">这让这些仓库中的日常工程工作变得更明确、更可靠。小改进的发布也更快了，验证、发布审查和 PR 交接都走同一套流程。</span></p><figure><span leaf=""><img alt="alt" data-ratio="0.5574074074074075"
