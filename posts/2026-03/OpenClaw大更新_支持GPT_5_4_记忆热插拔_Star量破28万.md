OpenClaw大更新：支持GPT-5.4，记忆热插拔，Star量破28万 by cnbeta
------
<div style="margin-top:10px" class="content" id="artibody"><p><strong>“我们没有小发布。”</strong>本周日，最热门的智能体项目 OpenClaw 发布了 2026.3.7 版本，引入了上下文引擎插件、lossless-claw 等功能，同时也更新了一系列新模型的支持。</p><div class="article-global"></div><p><img src="https://static.cnbetacdn.com/article/2026/0309/5e669b9ee5ce234.jpg"><br></p><p>从 GitHub 的更新日志上看，OpenClaw 2026.3.7 的更新内容多而复杂。不过总的来说 OpenClaw 目前的主线已经逐渐清晰：<strong>它正在将 OpenClaw 从一个实验性的智能体框架，推向一个具备高度持久性、可扩展性和生产级安全性的成熟 Agent 操作系统。</strong></p><p>具体来看，此次更新主要包括以下几个方面：</p><ul class=" list-paddingleft-2"><li><p><strong>智能体 / 上下文引擎插件接口：</strong>新增 Context Engine 插件插槽，支持完整的生命周期钩子。引入基于插槽的注册表和配置驱动的解析机制；添加 LegacyContextEngine 包装器以保留现有的上下文压缩行为；通过 AsyncLocalStorage 为插件运行时提供作用域子智能体运行时，并新增 sessions・get 网关方法。这使得类似 lossless-claw 的插件提供替代的上下文管理策略，而无需修改核心压缩逻辑，而当未配置该插件时，行为无变化。</p></li><li><p><strong>ACP / 持久化频道绑定：</strong>新增耐用的 Discord 频道和 Telegram 话题绑定存储，包括路由解析，以及 CLI / 文档支持，使 ACP 线程目标在服务重启后依然保持绑定，并能进行一致性管理。</p></li><li><p><strong>Telegram/ACP 话题绑定：</strong>在 /acp spawn 命令中支持 Telegram Mac Unicode dash 前缀；支持 Telegram 话题线程绑定（--thread here|auto）；将已绑定话题的后续回复自动路由至 ACP 会话；新增 Telegram 审批按钮，使用带有前缀的 approval-id 解析，且成功绑定后自动在话题内置顶确认信息。</p></li><li><p><strong>Telegram / 话题智能体路由：</strong>在论坛组和私聊话题中支持针对每个话题的 agentId 覆盖，使特定话题可以路由到具有隔离会话的专用智能体。</p></li><li><p><strong>新手引导 / 联网搜索：</strong>在配置向导中增加供应商选择步骤和完整的供应商列表，并在引导过程中支持 SecretRef 引用模式。</p></li><li><p><strong>工具 / 联网搜索：</strong>将 Perplexity 供应商切换为 Search API，支持提供结构化结果，并新增语言 / 地区 / 时间过滤器，使得搜集信息更精准。</p></li><li><p><strong>网关：</strong>为 gateway.auth.token 增加 SecretRef 支持，并加入身份验证模式防护栏。</p></li><li><p><strong>插件 / 系统上下文字段：</strong>新增 prependSystemContext 和 appendSystemContext 钩子，允许将插件的静态引导信息放置在系统提示词空间，以利用供应商缓存并降低重复提示词的 token 成本。</p></li><li><p><strong>新增对 google/gemini-3.1-flash-lite-preview 的原生支持，</strong>包括模型 ID 正规化、默认别名、多媒体理解图像查找、Google Gemini CLI 向前兼容回退及文档。</p></li><li><p>工具链与本地模型生态优化方面，在 2026.3.7 版本上，OpenClaw 重构了 Web 搜索工具，优化了 Ollama 等本地模型的处理机制，以及 Docker 容器部署。新版本完美兼容了 Ollama 等本地模型在“思考 / 推理（Thinking/Reasoning）”阶段的流式输出隔离，不再会将模型底层的思考过程误当做正式回复输出。</p></li></ul><p>此外在容器化上，Docker 支持通过 OPENCLAW_EXTENSIONS 预打包扩展依赖，容器启动更快速、结果更可复现。</p><p>需要注意的是，<strong>此次更新中一个重要变化是“重大”网关认证模式，</strong>如果同时配置 gateway.auth.token 和 gateway.auth.password，则现在必须显式指定 gateway.auth.mode，升级前将该模式设置为 token 或 password。否则，升级后可能导致启动、配对或 TUI 失败。</p><p>而这些只是此次更新的部分内容，更多详细信息，大家可以去 GitHub 上获取。正如 OpenClaw 在更新贴文中所言，<strong>“此次发布的背后包含了 196 位贡献者”，</strong>涉及多方面的更新。</p><p><img src="https://static.cnbetacdn.com/article/2026/0309/1a56de1981e92ae.jpg"><br></p><p>总结起来，OpenClaw 正在试图解决目前 AI Agent 框架工程化落地的几个重要问题。它突破了“上下文窗口”的瓶颈，允许开发者自由挂载 RAG（检索增强生成）、知识图谱折叠或无损压缩算法，为智能体“长期自主运行”铺平了道路。</p><p>在 OOLONG 基准测试中，使用相同的模型，<strong>新版 OpenClaw 上 lossless-claw 的得分为 74.8，而 Claude Code 的得分为 70.3，</strong>随着上下文长度的增加，差距会越来越大。</p><p>在所有测试的上下文长度下，它的性能都优于 Claude Code。有工程师在 OpenClaw 上运行了一周，表示：<strong>“说它运行良好都算是保守的说法。”</strong></p><p>OpenClaw 强化了主从智能体的协作能力，复杂任务现在可以更加安全、隔离地拆解并分发给拥有专属记忆通道的小智能体来处理。通过让不同的 Agent“常驻”在不同部门群组中的方法，OpenClaw 实现了持久化的通道绑定。现在哪怕服务器重启，专用的智能体依然能记住上下文并接续之前的工作。</p><p>最后，诸如防提示词注入、拦截、凭据保护、时间戳的 UTC 锚定对齐等更新进一步提升了安全性。</p><p>目前，<strong>OpenClaw 在 GitHub 上的 Star 量已经达到了 28 万。</strong></p><p><img src="https://static.cnbetacdn.com/article/2026/0309/764c26bec3ece11.jpg"><br></p><p>从社交媒体到办公室，越来越多的人正在尝试使用 OpenClaw，该工具的应用范围已经从专业的开发者转向各行各业的不同人群。人们整将其用于从选股、撰写报告到制作 PPT、写电子邮件、写代码等各种用途。</p><p>AI 智能体助手，或许会很快成为每个人手下的“虚拟员工”，大大降低我们的工作量。</p><p><strong>One More Thing</strong></p><p>机器之心拟于近期在上海举办一场 OpenClaw 活动，正在四处“抓”有趣的议题和嘉宾。<strong>这次想把 OpenClaw 彻底聊透，从爆红机制、技术逻辑，到实战案例、创业机会等，</strong>我们都想听。</p><p>特别欢迎<strong>OpenClaw 核心贡献者、大厂 / 科技公司团队负责人、实验室 / 高校学者、Agent 创业者、VC 投资人，以及在 OpenClaw 上有深入实践的朋友</strong>来聊。</p><p>有意者请联系小助手微信（微信号：13661489516）</p><p>参考内容：</p><p>https://x.com/openclaw/status/2030522386894946620</p><p>https://github.com/openclaw/openclaw/releases/tag/v2026.3.7</p></div>
