Claude Code 省 Token 指南：慎用 1M 上下文，不开新会话或者总是开新会话都不对 by 宝玉AI
------
<div><section><p><span leaf="">最近社区里怨声载道：配额烧得太快了。Max 用户一周的额度，有人两天就用完。有人在 AWS Bedrock 上算了一笔账，</span><strong><span leaf="">一个会话的真实成本超过 134 美元</span></strong><span leaf="">，而 Pro Max 5x 订阅一个月才 100 美元。Anthropic 的订阅本身就是亏本在卖。</span></p><p><span leaf="">很多人的第一反应是频繁 /clear，或者每做完一步就开新会话。觉得这样“轻装上阵”能省钱。但如果你理解了 Claude Code 背后的缓存机制，会发现这个操作经常适得其反：</span><strong><span leaf="">你刚刚触发了一次全价上下文重建，花的钱比继续聊下去还多。</span></strong></p><p><span leaf="">要理解配额为什么烧得快，得先搞清楚大语言模型处理输入的方式——它跟人类的“接着读”完全不同。</span></p><p><span leaf="">想象你在写一篇关于气候变化的论文，每次去图书馆都要做同一件事：找到那本 500 页的《全球气候报告》，翻到需要的章节，把关键数据抄下来。第一次花 40 分钟，第二次问了个不同的问题，还是同一本书，又花 40 分钟。第三次、第四次，同样的 40 分钟。</span></p><p><span leaf="">大语言模型干的就是这件事。它没有“记忆”，不会因为刚读过就跳过。每次收到你的消息，它都要从头“读”一遍完整的输入内容，读的时间和内容长度成正比。</span></p><p><span leaf="">在 Claude Code 里，输入内容通常包括三部分：</span></p><ul><li><section><span leaf="">• </span><strong><span leaf="">固定部分</span></strong><span leaf="">：系统指令、工具定义、CLAUDE.md 里的项目规则</span></section></li><li><section><span leaf="">• </span><strong><span leaf="">对话历史</span></strong><span leaf="">：之前所有的对话轮次</span></section></li><li><section><span leaf="">• </span><strong><span leaf="">新消息</span></strong><span leaf="">：你刚刚输入的那句话</span></section></li></ul><p><span data-heading="true"><span leaf="">提示缓存：把“笔记”存起来</span></h2><p><span leaf="">每次都从头翻那本 500 页的书，任谁都受不了。解决办法很直接：把笔记提前做好，下次只翻笔记本。</span></p><p><span leaf="">提示缓存干的就是这件事。</span></p><p><span leaf="">模型读完一段输入后，会生成一组中间计算结果，相当于“阅读笔记”。后续生成回答时，模型靠的就是这组笔记，不会再回去看原文。提示缓存的机制是：第一次算完笔记后存下来，下次再遇到相同的输入前缀，直接用存好的笔记，跳过重复计算。</span></p><p><strong><span leaf="">读取缓存的成本只有重新计算的十分之一。</span></strong></p><p><span leaf="">粗略算一下：一个 20 轮的会话，10 万 Token 的上下文，缓存一直命中的话，输入成本比每次都全价处理低 6 倍以上。</span></p><p><span leaf="">但缓存有两个前提条件。</span></p><p><strong><span leaf="">第一，缓存只对“前缀”有效。</span></strong><span leaf=""> 必须从头开始、一字不差地匹配。</span></p><p><span leaf="">打个比方：你在作文纸上写了一篇文章，前 3 页一字不差，第 4 页改了一个字。只有前 3 页能用缓存，第 4 页开始就得重新计算。如果你在第 1 页就改了一个字？整个缓存全部失效，从头算。</span></p><p><span leaf="">所以 Claude Code 的输入结构很重要：系统指令、工具定义这些不变的内容放在最前面，对话历史在中间，新消息放在最后。每次只有末尾那一小段需要重新计算，前面一大段都能命中缓存。</span></p><p><span leaf="">在同一个活跃会话里，前缀天然一致，每一轮只是在尾部追加新内容，缓存命中率很高。但如果你开了一个新会话，前缀从零开始，之前积累的缓存全部用不上。</span></p><section nodeleaf=""><img data-heading="true"><span leaf="">三个反直觉的省钱策略</span></h2><p><span leaf="">理解了缓存之后，有些“常识”要翻过来。</span></p><h3 data-heading="true"><span leaf="">缓存还热的时候，继续聊比开新会话便宜</span></h3><p><span leaf="">Claude Code 每次新会话启动，都要重新加载系统提示、工具定义、CLAUDE.md、项目配置。这些“基础设施”大约 5 万 Token。</span><strong><span leaf="">频繁 /clear 等于反复为这些不变的内容付全价写入费。</span></strong></p><p><span leaf="">而在活跃会话里，这些内容一直在缓存中，每次只付十分之一的价格。</span></p><p><span leaf="">Anthropic 员工 Lydia Hallie 说的“闲置约一小时的大型会话，建议重新开始”，关键词是“闲置”。活跃工作中的会话，缓存一直是热的，继续聊才最省。</span></p><h3 data-heading="true"><span leaf="">复杂任务一次做对，比来回改三轮更省</span></h3><p><span leaf="">关掉扩展思考确实能在单次请求里省 Token。但一个复杂的重构任务，开着扩展思考一次搞定，和关掉之后来回改三轮，</span><strong><span leaf="">后者更贵的概率很大</span></strong><span leaf="">。因为每多一轮对话，整个上下文都要重新发送一次，三轮累积的 Token 远超一次深度思考的额外开销。</span></p><p><span leaf="">简单任务反过来。把 /effort 调低或者在 /config 里关掉思考模式，效果立竿见影。思考 Token 按输出价计费，默认预算对简单任务来说浪费明显。</span></p><h3 data-heading="true"><span leaf="">长内容给路径，别往对话里贴</span></h3><p><span leaf="">比起控制输出长度，更有效的是控制输入质量。不要把 10000 行日志复制粘贴到对话里让 Claude 自己找错误，直接把日志文件路径发给它。Claude Code 会自己用 grep 之类的工具去检索需要的信息，只把相关内容拉进上下文。</span><strong><span leaf="">最便宜的 Token，永远是根本没进上下文的 Token。</span></strong></p><section nodeleaf=""><img data-heading="true"><span leaf="">继续聊还是开新会话：一张决策表</span></h2><p><span leaf="">这可能是 Claude Code 省 Token 最关键的一个判断。很多人的默认习惯是“做完就清”，实际上最省的默认习惯应该反过来：</span><strong><span leaf="">能继续就继续，开新会话是有条件触发的操作。</span></strong></p><p><span leaf="">满足以下任一条件，</span><strong><span leaf="">继续当前会话</span></strong><span leaf="">：</span></p><ul><li><section><span leaf="">• 任务没变。还在调同一个 bug、写同一个模块、围绕同一组文件工作。</span></section></li><li><section><span leaf="">• 距离上一条消息不超过 1 小时。缓存还活着，前面积累的上下文几乎不花钱。</span></section></li><li><section><span leaf="">• 上下文里的内容对当前工作仍然有用。之前读过的文件、讨论过的方案，模型还在用。</span></section></li></ul><p><span leaf="">如果在思考问题暂时没有输入，可以发一条简短消息保持缓存活跃。有用户甚至写了心跳扩展来自动保活缓存，刷新一次缓存的成本只有完整缓存未命中的十分之一。</span></p><p><span
