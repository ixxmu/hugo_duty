leaf="">上次做错了什么</span></h2><p><span leaf="">上次的问题不是技术路线选错了，也不是工具不好用。问题出在我的做事方式上。</span></p><p><strong><span leaf="">我把"完美设计"当成了第一优先级。</span></strong><span leaf=""> DESeq2 的实现细节很多——IRLS 收敛条件、Cox-Reid adjustment、经验贝叶斯 prior、Cook's distance、independent filtering。正因为知道它复杂，我更倾向于一开始就把标准拉满：架构要干净，验证要精确到 bit，每个中间步骤都要有 R 的 ground truth 做 oracle。</span></p><p><strong><span leaf="">我建了一堆基础设施，但没有先把核心跑通。</span></strong><span leaf=""> 我写了 oracle fixture 导出工具、field-level comparison 框架、worktree 管理脚本、多阶段 verify 流程。这些东西单独看都对，但它们在主问题尚未收敛时同时出现，效果很糟糕：我持续获得"我在推进"的幻觉，却迟迟拿不到一个能跑通的 </span><code><span leaf="">deseq2_run()</span></code><span leaf=""> 调用。</span></p><p><strong><span leaf="">我用复杂度来回避不确定性。</span></strong><span leaf=""> 因为对 IRLS 的数值行为不够确定，对 dispersion fitting 的收敛路径不够确定，所以我本能地去多做几层保证——更完整的 plan、更细的 task 拆分、更多的中间验证。这些动作表面上在降低风险，实际上在推迟面对核心问题的时间。</span></p><h2 data-heading="true"><span leaf="">这次做对了什么</span></h2><h3 data-heading="true"><span leaf="">1. 定义了一个极其具体的最小目标</span></h3><p><span leaf="">不是"用 Rust 重写 DESeq2"。而是：</span></p><blockquote><p><span leaf="">在 airway 数据集上，用 </span><code><span leaf="">padj &lt; 0.01</span></code><span leaf=""> 和 </span><code><span leaf="">|log2FC| &gt; 1</span></code><span leaf=""> 筛选基因，Rust 和 R 的结果要一样。</span></p></blockquote><p><span leaf="">就这一个目标。不是 bit-to-bit 精确。不是功能完整。不是架构优雅。就是：</span><strong><span leaf="">两个实现在同一个数据集上同意哪些基因是差异表达的。</span></strong></p><p><span leaf="">这个目标具体到可以用一个 Python 脚本在 10 秒内验证。每次改完代码，跑一遍就知道离目标还有多远。</span></p><h3 data-heading="true"><span leaf="">2. 砍掉了所有非核心功能</span></h3><p><span leaf="">DESeq2 有很多功能。这次我只实现了最小 pipeline：</span></p><ul><li><section><span leaf="">• Size factor estimation（median-of-ratios）</span></section></li><li><section><span leaf="">• Dispersion estimation（gene-wise MLE → parametric trend → MAP shrinkage）</span></section></li><li><section><span leaf="">• Negative binomial GLM fitting（IRLS）</span></section></li><li><section><span leaf="">• Wald test</span></section></li><li><section><span leaf="">• BH p-value adjustment + independent filtering</span></section></li></ul><p><span leaf="">砍掉的：LRT、lfcShrink（apeglm/ashr/normal）、formula 解析、tximport、VST、rlog、outlier replacement、Cook's distance。</span></p><p><span leaf="">这些都是重要功能。但在核心 pipeline 跑通之前，它们都是噪音。</span></p><h3 data-heading="true"><span leaf="">3. 逐步验证，每一步都和 R 对比</span></h3><p><span leaf="">我写了一个 R 脚本，把 DESeq2 在 airway 数据上每一步的中间结果都导出成 TSV：</span></p><pre><span><svg xmlns="http://www.w3.org/2000/svg" version="1.1" x="0px" y="0px" width="45px" height="13px" viewbox="0 0 450 130"><ellipse cx="50" cy="65" rx="50" 输出 509 个显著基因而 R 输出 675 个时，我没有开始重写架构。而是逐步排查：</span></p><ul><li><section><span leaf="">• Size factors 对不对？ → 完美匹配（误差 &lt; 1e-10）</span></section></li><li><section><span leaf="">• Gene-wise dispersions 对不对？ → 一半基因误差 &gt;10%</span></section></li><li><section><span leaf="">• 为什么？ → </span><strong><span leaf="">发现 bug：传了 normalized counts 给 NB log-posterior，应该传 raw counts</span></strong></section></li></ul><p><span leaf="">修了这个 bug 之后：</span></p><ul><li><section><span leaf="">• Gene-wise dispersions 对不对？ → 99.7% 基因误差 &lt;1%</span></section></li><li><section><span leaf="">• 但还有 4400 个 outlier → </span><strong><span leaf="">加了 all-zero 基因过滤、moments 估计、收敛检查</span></strong></section></li></ul><p><span leaf="">修了之后 overlap 从 0% 涨到 99.6%。每一步修复都有明确的诊断依据，不是猜。</span></p><h3 data-heading="true"><span leaf="">5. 知道什么时候该停</span></h3><p><span leaf="">最终 overlap 停在 672/675 = 99.6%。剩下的 3 个 R-only 基因都是 borderline case（R padj = 0.008-0.009，Rust padj = 0.010-0.012）。39 个 Rust-only 基因来自 dispersion trend 系数的 ~5% 偏差。</span></p><p><span leaf="">要关掉这 0.4% 的差距，需要在 dispersion estimation 里实现 R 的完整 NB GLM mu 迭代（R 用 </span><code><span leaf="">fitNbinomGLMs</span></code><span leaf=""> 计算 mu，我们用的是 linear model 投影）。这是一个确定的技术任务，但工作量不小，对 MVP 来说不值得。</span></p><p><span leaf="">所以我停了。记录下 root cause，写进 TODO，提交，推送。</span></p><p><span leaf="">上一次我在类似的岔路口选择了"继续做到完美"，然后在完美的路上耗尽了所有预算。这一次我选择了"够好就发"。</span></p><h2 data-heading="true"><span leaf="">架构长什么样</span></h2><p><span leaf="">最终选了 Functional Core + Pipeline Shell 的架构：</span></p><p><strong><span leaf="">Functional Core</span></strong><span leaf="">：每个算法步骤是纯函数，输入明确，输出明确，可以独立测试和对比。这让逐步验证变得非常自然——</span><code><span leaf="">estimate_size_factors()</span></code><span leaf=""> 的输出直接和 R 的 </span><code><span leaf="">sizeFactors()</span></code><span leaf=""> 对比，不需要跑整条 pipeline。</span></p><p><strong><span leaf="">Pipeline Shell</span></strong><span leaf="">：</span><code><span leaf="">DESeqDataSet</span></code><span leaf=""> 结构体串联所有纯函数，提供 </span><code><span leaf="">run()</span></code><span leaf=""> 和 </span><code><span leaf="">results()</span></code><span leaf=""> 方法。CLI 包一层 clap 就完了。</span></p><p><span leaf="">依赖全部是纯 Rust：</span><code><span leaf="">faer</span></code><span leaf="">（线性代数），</span><code><span leaf="">rayon</span></code><span leaf="">（并行），</span><code><span leaf="">statrs</span></code><span leaf="">（统计函数）。不需要系统 BLAS/LAPACK，编译即用，跨平台友好。</span></p><h2 data-heading="true"><span leaf="">给同行的建议</span></h2><p><span leaf="">如果你也在用 AI 做科学计算软件的重构或重写：</span></p><p><strong><span leaf="">先定义最小可验证闭环。</span></strong><span leaf=""> 对于 DESeq2 来说就是"一个数据集、一组参数、比较输出"。对于你的项目，找到等价的东西。所有工作都围绕这个闭环展开。</span></p><p><strong><span leaf="">先让结果大致对，再追求精确对。</span></strong><span leaf=""> 99% 的 overlap 比 0% 的 bit-to-bit 就算错了。</span></p><p><strong><span leaf="">知道什么时候该停。</span></strong><span leaf=""> 0.4% 的精度差距有明确的 root cause 和修复路径。但修复它的 ROI 不足以支撑在 MVP 阶段投入。记录下来，留给下一次。</span></p><p><strong><span
